# Flink idioms (1.20 LTS / 2.3; config keys common to both)

## Exactly-once honesty — the #1 consulting conversation

Flink's "exactly-once" is exactly-once **STATE** semantics (checkpoint barriers +
rewind-and-replay). End-to-end exactly-once additionally requires:

- **Replayable sources** (Kafka: offsets live in checkpoint state), AND
- **Transactional (2PC) sinks** — or idempotent writes. Flink "cannot revert writes to
  external systems": on replay, duplicates are harmless for idempotent K/V upserts but
  real for append logs unless the sink is transactional.

"We enabled exactly-once but the destination has duplicates" is almost always a
NONE/at-least-once sink, not a Flink bug (see Kafka sink default below).

## Checkpoint configuration that matters

`execution.checkpointing.*`: `interval` (drives sink visibility latency — exactly-once
sinks only emit on checkpoint completion), `timeout`, `min-pause` (guarantees progress),
`tolerable-failed-checkpoints` (**default 0** — first failure fails the job over; decide
deliberately), `externalized-checkpoint-retention`, `incremental` (RocksDB only),
`storage: filesystem` + `dir` on durable storage (**JobManager-heap default is not
production**).

**Aligned vs unaligned**: under backpressure, alignment dominates checkpoint time.
Remedies in order: fix the backpressure → buffer debloating
(`taskmanager.network.memory.buffer-debloat.enabled`) → unaligned checkpoints (barriers
overtake in-flight data; duration becomes throughput-independent, but bigger checkpoints
and extra state I/O). `aligned-checkpoint-timeout` gives the adaptive middle path.
Savepoints are always aligned.

## Watermarks and late data

- Prefer `WatermarkStrategy` ON the source (per-partition watermarking);
  `forBoundedOutOfOrderness` tuned to measured disorder.
- **Operator watermark = min over inputs** — one idle partition stalls everything
  downstream. `.withIdleness(Duration)` is the fix; Kafka does NOT auto-idle when
  parallelism > partition count (right-size or set idleness).
- **Late data is DROPPED silently by default** (`allowedLateness` = 0). Either capture
  it (`sideOutputLateData` — the de-core no-silent-drops rule) or accept the loss
  explicitly. `allowedLateness(t)` re-fires windows → downstream receives UPDATED
  results (duplicates unless it upserts). Detection metric: `numLateRecordsDropped`.
- Watermark alignment (`withWatermarkAlignment`) throttles fast sources to bound join/
  window state.

## State

- Backends: `hashmap` (heap, fast, memory-bounded — set managed memory ~0) vs `rocksdb`
  (disk, only backend with incremental checkpoints, ~10x slower per access; lives in
  managed memory). 2.x adds experimental disaggregated `forst` — watch, don't default.
- **State TTL** is processing-time only, per-entry, and expired state persists if never
  accessed and no background cleanup fires — configure the cleanup strategy matching the
  backend. 1.20: flipping TTL on existing state breaks restore (2.2+ migrates
  seamlessly).
- **Schema evolution**: POJO and Avro only. Keys can NEVER evolve; anything serialized
  by Kryo is frozen forever (a `List<OtherPojo>` field inside a POJO → Kryo → frozen) —
  review serialization before the first production savepoint, not after.

## Savepoints vs checkpoints (the ops contract)

- Checkpoints = recovery; savepoints = planned ops (upgrade, rescale, fork, backend
  change). **Only canonical savepoints support changing the state backend.**
- **`uid()` on every stateful operator** — auto-generated IDs are topology-sensitive;
  missing UIDs make savepoints unusable after refactors. Enforce with
  `disableAutoGeneratedUIDs`. Restore drops orphaned state only with `-n`.
- Claim modes: NO_CLAIM default (first checkpoint full, then the source snapshot is
  deletable) vs CLAIM. Intermediate savepoints don't commit side effects — never run two
  jobs off one checkpoint timeline.

## SQL/Table vs DataStream

- SQL for relational/CDC logic — know the three changelog encodings (append-only /
  retract / upsert-with-key) when wiring CDC flows.
- Two caveats: update-queries without windows keep state for every emitted row FOREVER
  unless TTL bounds it; and the planner owns the topology — **any query change or minor
  Flink upgrade may be state-incompatible**. DataStream where explicit state control and
  upgradability matter.

## Kafka connector gotchas (all doc-verbatim)

1. Source default start = `earliest`, NOT committed offsets; offsets are checkpoint
   state — broker-committed offsets are monitoring-only.
2. **Sink `DeliveryGuarantee` defaults to NONE.**
3. EXACTLY_ONCE sink: unique `transactionalIdPrefix` per application; consumers need
   `isolation.level=read_committed`; visibility latency = checkpoint interval; and set
   producer `transaction.timeout.ms` ≫ max checkpoint + restart duration **"or data loss
   may happen"** (broker aborts the uncommitted transaction — the
   `ProducerFencedException` incident).
